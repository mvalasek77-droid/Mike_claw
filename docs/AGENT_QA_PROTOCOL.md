# Systematic Agent QA Protocol

This protocol exists because ad hoc "looks good" reviews miss basic
bugs. Every non-trivial CodeGenie release, and every app generated by
CodeGenie, must be checked from an explicit page/process inventory.

The rule is simple: no final green check without evidence for every
reachable page, state, action, and release process.

## Required Artifact

Every build that changes user-facing behavior must create or update:

`docs/qa/PAGE_PROCESS_MATRIX.md`

The matrix is the contract between agents. It must list:

- Every reachable page, sheet, modal, tab, onboarding step, and empty state.
- Every user action on those surfaces: buttons, links, toggles, text entry,
  picker changes, destructive actions, purchase/restore, sharing, upload,
  retry, cancel, back navigation, and background/foreground resume.
- Every important state: first launch, returning user, signed out, signed in,
  no network, slow network, missing credentials, invalid credentials, loading,
  success, validation failure, hard failure, light mode, dark mode,
  Dynamic Type AX5, Reduce Motion, and VoiceOver.
- Expected result, visible user feedback, persistence expectation, and
  evidence required to pass.

Rows without evidence fail the release gate. "Not tested" is allowed as a
status, but it blocks final approval.

## Agent Lenses

Each agent checks the same matrix, but each one uses a different failure
model. Agents must not copy another agent's conclusion.

| Agent | Lens | Must Produce |
| --- | --- | --- |
| Architect | Inventory completeness | Page/process matrix with all surfaces, states, and actions |
| Designer | Visual and interaction quality | Light/dark/Dynamic Type/Reduce Motion notes per page |
| Coder | Implementation wiring | Confirmation that every action has a real handler and error path |
| Unit Tester | Logic contracts | Tests for non-view services, models, state machines, and failure modes |
| UI Tester | Real user traversal | Simulator evidence, screenshots, and XCUITest coverage for each row |
| Reviewer | Senior engineering skepticism | Findings for correctness, lifecycle, threading, persistence, and regressions |
| Security Auditor | Privacy and abuse cases | Secret handling, PII logging, URL/file/path/network permission review |
| Final Judge | Release gate | Independent sign-off that every matrix row has evidence and no blocker remains |

## Pass Order

1. Inventory pass
   - Architect creates `docs/qa/PAGE_PROCESS_MATRIX.md`.
   - If a surface can be reached from Home, Settings, a tab, a sheet, a push
     route, an error state, or a deep link, it must have rows.

2. Implementation pass
   - Coder and Designer implement against the matrix.
   - Any new surface or action updates the matrix in the same commit.

3. Independent verification pass
   - Unit Tester covers non-view logic.
   - UI Tester drives the simulator and attaches screenshots or XCUITest names.
   - Security Auditor checks every row that touches credentials, network,
     file paths, privacy, purchases, account state, or external URLs.
   - Reviewer reads the diff and the matrix, then tries to disprove the green
     result rather than confirming it.

4. Final Judge pass
   - Use the strongest available model for Reviewer and Security final passes.
   - Final Judge must fail the release if:
     - Any page/action row has no evidence.
     - A UI-changing diff does not update the matrix.
     - Light mode or dark mode has unreadable text.
     - A primary button has no real handler.
     - A loading state can hang without timeout, retry, or error feedback.
     - Pricing, privacy, Apple Developer, GitHub, TestFlight, or App Store
       copy overstates what is actually automated.
     - Any critical Reviewer or Security finding remains open.

## Evidence Standard

Every completed matrix row needs at least one evidence token:

- `test:<test_name>` for XCTest, pytest, or XCUITest.
- `screenshot:<path>` for simulator or device screenshots.
- `log:<command>` for build, upload, archive, or backend logs.
- `manual:<device/os/date>` for real-device steps that cannot be automated.
- `code:<file:line>` for static checks where runtime evidence is not useful.

Text such as "checked", "seems fine", "works", or "no issue" is not evidence.

## Mandatory Core Flows

For CodeGenie itself, the matrix must always include these flows:

- First launch: splash, onboarding, tutorial replay, terms/privacy gate.
- Home: sample build, start new build, recent project, automation audit,
  app-of-year playbook, settings entry, account/settings entry.
- Builder: prompt entry, model/auth preflight, cost confirmation, build start,
  pause, resume, snapshot, fork, retry, cost cap, failure overlay, success
  overlay, Perfection Mode, simulator preview, GitHub backup, App Store submit.
- Settings: BYOK keys, subscription pairing, hosted pricing, purchase,
  restore, free quota exhaustion, cost cap, per-agent routing, custom agents,
  Apple Developer setup, Mac pairing, GitHub setup, bug report, telemetry,
  admin/archive, changelog.
- Automation: Xcode readiness, App Store Connect guidance, privacy and terms
  setup, icon generation, screenshots, TestFlight upload, TestFlight polling,
  GitHub sync, release readiness.
- System states: fresh install, returning install, missing credentials,
  invalid credentials, offline, slow network, background resume, low storage,
  light mode, dark mode, AX5 Dynamic Type, Reduce Motion, VoiceOver.

## Merge Gate

Before code lands on `main`, the release owner must record:

- Backend tests result.
- Xcode build result and Xcode version.
- Simulator/device smoke result.
- Matrix coverage result: total rows, passed rows, blocked rows.
- Final Reviewer model and Final Security model used for sign-off.

If the matrix is missing, stale, or partially unevidenced, the correct final
answer is "not release ready" even when the app builds.
